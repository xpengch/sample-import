# 芯片样片问题定位系统 - 架构文档

## 目录结构

```
sample_import/
├── board_controller/          # 单板控制模块 (v6新增)
│   ├── __init__.py
│   ├── connections.py         # 双通道连接管理
│   ├── monitors.py            # 监控器(挂死检测+启动监控)
│   ├── resume.py              # 断点续传管理器
│   └── controller.py          # 统一控制器
├── mcp_server/                # MCP工具服务器
│   ├── tools.py               # 工具接口定义
│   ├── mock_tools.py          # Mock实现
│   └── server.py              # 服务器
├── skills/                    # 分析技能(11个)
│   ├── hypothesis-manager/    # 假设迭代管理器(元技能)
│   ├── test-planner/          # 测试规划技能
│   ├── clock-analysis/        # 时钟域分析
│   ├── power-analysis/        # 电源分析
│   ├── temp-analysis/         # 温度分析
│   ├── signal-analysis/       # 信号完整性分析
│   ├── memory-analysis/       # 内存分析
│   ├── hang-analysis/         # 挂死问题分析(v6支持)
│   ├── crash-analysis/        # 崩溃问题分析

│   └── general-diagnose/      # 通用诊断
├── core_utils/                # 核心工具
│   └── state_manager.py       # 状态管理器
├── config/                    # 配置文件
│   └── board_config.yaml      # 单板配置(v6新增)
├── tests/                     # 测试文件
└── docs/                      # 文档
```

## 2. 假设迭代管理器 (hypothesis-manager) 工作流程

### 七阶段迭代流程

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        Phase A: 初始分析 (Initial Analysis)                  │
├─────────────────────────────────────────────────────────────────────────────┤
│  1. 收集问题描述 (Problem Description)                                      │
│  2. 分析症状 (Symptom Analysis)                                             │
│  3. 识别关键线索 (Key Clues)                                                │
│  4. 确定分析方向 (Analysis Direction)                                       │
└─────────────────────────────────────────────────────────────────────────────┘
                                      │
                                      ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                        Phase B: 假设生成与评估 (Hypothesis Evaluation)       │
├─────────────────────────────────────────────────────────────────────────────┤
│  1. 生成潜在假设 (Generate Hypotheses)                                       │
│     - 基于症状的可能原因                                                    │
│     - 专业领域知识 (时钟/电源/温度/信号等)                                    │
│  2. 评估假设可测试性 (Evaluate Testability)                                 │
│     - 需要什么测试                                                          │
│     - 测试可行性                                                            │
│  3. 选择优先假设 (Select Priority Hypothesis)                               │
└─────────────────────────────────────────────────────────────────────────────┘
                                      │
                                      ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                        Phase C: 测试规划 (Test Planning)                     │
├─────────────────────────────────────────────────────────────────────────────┤
│  1. 调用 test-planner 技能                                                  │
│  2. 设计测试用例 (Design Test Cases)                                         │
│     - 验证假设的测试步骤                                                     │
│     - 预期结果定义                                                          │
│  3. 确定测试工具 (Select Test Tools)                                         │
│     - clock_test / power_test / temp_test / general_test                   │
│  4. 生成测试计划 (Generate Test Plan)                                        │
└─────────────────────────────────────────────────────────────────────────────┘
                                      │
                                      ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                        Phase D: 测试执行 (Test Execution)                    │
├─────────────────────────────────────────────────────────────────────────────┤
│  1. 调用 MCP 工具执行测试                                                    │
│     - board_control: 单板控制操作 (v6)                                       │
│     - clock_test: 时钟域测试                                                 │
│     - power_test: 电源测试                                                   │
│     - temp_test: 温度测试                                                    │
│     - general_test: 通用测试                                                 │
│  2. 收集测试数据 (Collect Test Data)                                         │
│  3. 处理特殊情况 (Handle Special Cases)                                      │
│     - 挂死检测与恢复 (v6)                                                     │
│     - 断点续传 (v6)                                                          │
└─────────────────────────────────────────────────────────────────────────────┘
                                      │
                                      ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                        Phase E: 结果分析 (Result Analysis)                   │
├─────────────────────────────────────────────────────────────────────────────┤
│  1. 分析测试结果 (Analyze Test Results)                                      │
│  2. 假设验证 (Hypothesis Verification)                                        │
│     - 支持假设: 继续深入或生成报告                                           │
│     - 拒绝假设: 生成新假设                                                   │
│     - 不确定: 设计更多测试                                                   │
│  3. 更新状态 (Update State)                                                  │
│     - 保存测试结果到 StateManager                                            │
│     - 更新假设列表                                                           │
└─────────────────────────────────────────────────────────────────────────────┘
                                      │
                                      ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                        Phase F: 继续决策 (Continue Decision)                 │
├─────────────────────────────────────────────────────────────────────────────┤
│  检查是否需要继续迭代:                                                       │
│  ┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐       │
│  │ 假设已验证       │     │ 需要更多测试     │     │ 假设被拒绝       │       │
│  │ → 生成报告       │     │ → 返回 Phase C   │     │ → 返回 Phase B   │       │
│  │   (Phase G)      │     │   规划新测试     │     │   生成新假设     │       │
│  └─────────────────┘     └─────────────────┘     └─────────────────┘       │
└─────────────────────────────────────────────────────────────────────────────┘
                                      │
                                      ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                        Phase G: 报告生成 (Report Generation)                 │
├─────────────────────────────────────────────────────────────────────────────┤
│  1. 汇总分析过程 (Summarize Analysis Process)                                 │
│  2. 列出验证的假设 (List Verified Hypotheses)                                 │
│  3. 提供测试证据 (Provide Test Evidence)                                      │
│  4. 生成诊断报告 (Generate Diagnostic Report)                                 │
│  5. 保存状态 (Save State)                                                    │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 各阶段详细说明

#### Phase A: 初始分析
```
输入: 用户问题描述
输出: 症状分析、关键线索、分析方向

操作:
  1. 解析用户输入的问题描述
  2. 识别问题类型 (挂死/崩溃/性能/功能异常等)
  3. 提取关键症状
  4. 确定需要调用哪个专业分析技能
```

#### Phase B: 假设生成与评估
```
输入: 症状分析结果
输出: 优先假设列表

操作:
  1. 基于症状生成可能的根因假设
  2. 对每个假设评估:
     - 可测试性: 是否可以设计测试验证
     - 优先级: 基于概率和影响
  3. 选择最高优先级假设进行验证
```

#### Phase C: 测试规划
```
输入: 待验证假设
输出: 测试计划

操作:
  1. 调用 test-planner 技能
  2. 设计验证假设的测试用例
  3. 确定需要的测试工具
  4. 生成详细的测试步骤
```

#### Phase D: 测试执行
```
输入: 测试计划
输出: 测试结果

操作:
  1. 通过 MCP 调用相应测试工具
  2. 对于挂死问题 (v6):
     - 检测单板挂死状态
     - 必要时重启单板
     - 等待启动完成
     - 使用断点续传恢复测试状态
  3. 收集测试数据
  4. 保存测试结果到 StateManager
```

#### Phase E: 结果分析
```
输入: 测试结果
输出: 假设验证结论

可能的结论:
  1. 支持: 测试结果支持假设
     → 可选择生成报告或继续深入分析
  2. 拒绝: 测试结果与假设矛盾
     → 返回 Phase B 生成新假设
  3. 不确定: 测试结果不足以判断
     → 返回 Phase C 设计更多测试
```

#### Phase F: 继续决策
```
输入: 假设验证结论
输出: 下一步行动或报告

决策逻辑:
  IF 假设得到充分支持 AND 达到迭代上限:
    → 进入 Phase G 生成报告
  ELSE IF 假设被拒绝:
    → 返回 Phase B 生成新假设
  ELSE IF 需要更多证据:
    → 返回 Phase C 规划新测试
```

#### Phase G: 报告生成
```
输入: 完整迭代历史
输出: 诊断报告

报告内容:
  1. 问题描述
  2. 分析过程 (迭代历史)
  3. 验证的假设
  4. 测试证据
  5. 根因结论
  6. 建议措施
```

## 10. 使用示例

### 基础使用

```python
from board_controller import BoardController

# 配置
config = {
    "mode": "local",
    "serial": {
        "port": "COM3",
        "baudrate": 115200
    },
    "ping": {
        "host": "192.168.1.100"
    },
    "boot": {
        "keywords": ["login:", "root@"]
    }
}

# 创建控制器
controller = BoardController(config)
await controller.initialize()

# 重启单板并等待启动
result = await controller.restart_board(
    restart_type="soft",
    wait_for_boot=True,
    timeout=300
)

if result["success"]:
    print("单板重启成功")
else:
    print(f"重启失败: {result['error']}")
```

### 挂死问题分析流程

```python
# 1. 检测挂死
is_hung, details = await controller.detect_hang(
    detection_time=60,
    strict_mode=True
)

if is_hung:
    print("检测到单板挂死")
    print(f"  串口活动: {details['serial_activity']}")
    print(f"  Ping成功: {details['ping_success']}")

    # 2. 保存检查点
    controller.save_checkpoint("before_restart", {
        "detected_hang": True,
        "details": details
    })

    # 3. 重启单板
    result = await controller.restart_board(
        restart_type="hard",  # 挂死后需要硬重启
        wait_for_boot=True,
        timeout=300
    )

    # 4. 验证恢复
    if result["success"]:
        controller.save_checkpoint("after_restart", {
            "status": "recovered",
            "timestamp": datetime.now().isoformat()
        })
```
